# 컴퓨터응용교육특론

## 1. 개요 (Course Overview)
- **내용**: 바이브코딩을 활용하여 교육현장에 필요한 프로젝트 개발 (Vibe Coding / Antigravity)
- **대상**: 교육대학원 컴퓨터응용교육특론 수강자
- **운영**: 7회 수업 + 기말 종합평가(최종 발표/시연)
- **핵심 목표**:
  1) Vibe Coding을 통해 **자연어로 설계하고, AI와 함께 구현·개선하는 개발 경험**을 체득
  2) 교육 현장에서 즉시 활용 가능한 개인 맞춤형 프로그램(1개)을 기획·구현·배포
  3) 도구 활용(Notion/GitHub/AI Agent)을 통해 기획–개발–문서화–공유(배포)

## 2. 수업 산출물

### (A) 최종 결과물
- 교육 현장 적용 가능한 프로그램 1개 (예시)
  - 수업/동아리/프로젝트용 **웹 페이지**
  - **채점/평가 보조 프로그램**(CSV/엑셀 입력, 루브릭 자동화, 피드백 생성 등)
  - **퀴즈/문항 생성 및 관리 도구**(난이도/유형별, 출력/배포 기능)
  - 수업 운영 자동화(출석/과제 알림/결과 집계/리포트 등)
- 실행 가능한 형태로 제공(웹 링크 또는 실행 방법 포함)

### (B) 문서
- **md(마크다운 문서)**: 목적, 사용자(교사/학생), 주요 기능, 실행/사용 방법, 예시 화면(스크린샷)을 깃허브 페이지에 제공
- **수업 적용 시나리오**: “언제/누구에게/어떤 흐름으로/어떤 효과”를 명시
- **개발 로그(간단)**: 프롬프트/결정/이슈/해결 기록(노션 또는 md)

## 3. 활용 도구 (Tools)
- **Notion**: 기획서, 일정/작업관리, 개발 로그, 포트폴리오(최종 제출 페이지)
- **GitHub**: 공유, 배포(GitHub Pages)
- **Antigravity (AI Agent)**: 코드 생성/디버깅/리팩토링/페어프로그래밍

> 수업 원칙: “완벽한 이론”보다 **작동하는 프로그램**를 우선한다.

## 4. 운영 방식
매 회차 공통 루틴:
1) 문제/요구 정의(교육 맥락)  
2) 프롬프트 설계(입력/출력/제약조건 명시)  
3) 생성된 코드 실행 및 오류 재현  
4) 디버깅/개선(기능/UX/예외처리)  
5) 기록(노션/깃 커밋/스크린샷)

## 5. 회차별 강의 계획

| 회차 | 주제                       | 핵심 내용                                                   | 수업 내 산출물·과제                          |
| ----- | ------------------------ | ------------------------------------------------------- | ------------------------------------ |
| 1 | 오리엔테이션 + Vibe Coding 이해  | 수업 목표·평가 안내, Vibe Coding 개념, AI로 프로그램 만드는 흐름 이해, 간단한 시연 | (과제) 교육 현장에서 만들고 싶은 프로그램 아이디어 3가지 정리, 깃허브, 노션 가입하기 |
| 2 | Vibe Coding 실습 (미니 프로그램) | 자연어로 기능 요청하기, AI가 만든 코드 실행·수정 경험, 간단한 프로그램 완성           | (과제) 미니 프로그램에 기능 1개 추가 + 간단 설명 작성    |
| 3 | 데이터·파일 활용하기              | CSV/엑셀/텍스트 파일을 활용한 기능 만들기, 채점·집계·퀴즈의 기본 구조 이해           | (과제) 파일을 입력으로 받아 결과를 출력하는 기능 구현      |
| 4 | 수업 적용 관점에서 다시 설계하기       | 실제 수업·업무 흐름에 맞게 프로그램 구조 정리, 화면 구성과 사용 흐름 점검             | (과제) 수업 적용 시나리오 1페이지 + 화면 예시         |
| 5 | 핵심 기능 만들기                | 내 프로그램에서 **가장 중요한 기능** 구현, 실제로 실행되는 상태까지 완성             | (과제) 핵심 기능 실행 화면 캡처 + 중간 결과 정리       |
| 6 | 사용 방법 정리 및 공유 준비         | 오류 수정, 사용 방법 설명 작성, 다른 교사도 사용할 수 있도록 정리                 | (과제) README 최종본 + 실행 링크 또는 실행 방법     |
| 7 | 최종 점검 및 발표 준비            | 결과물 점검, 서로 테스트해보기                            | (과제) 최종 제출물 정리                       |

## 6. 기말평가(최종 발표/시연)
- **형식**: 1인(또는 2인 팀 선택 가능) 시연 중심 발표
- **권장 구성(7~10분)**:
  1) 교육 현장 문제/니즈(1분)
  2) 솔루션 핵심 기능(2분)
  3) 라이브 데모(3분)
  4) 적용 시나리오/운영 방법(1~2분)
  5) 한계/개선 계획(1분)

## 7. 평가 방법 (Grading)
> 결과물 “완성도”뿐 아니라, **교육 현장 적용 가능성/재현성/신뢰성**을 중점 평가한다.

- **출석 및 참여(20%)**
  - 수업 내 실습 수행, 피드백 반영, 로그/커밋 성실도

- **중간 점검(20%)**  *(4~5회차 산출물 기반)*
  - 문제정의/대상 명확성(10)
  - 진행도(10)
  - 적용 시나리오/UX 설계(10)

- **기말 프로젝트(60%)**
  - 기능 완성도 및 안정성(20) : 핵심 기능 작동, 예외처리, 오류 메시지
  - 교육적 유용성(15) : 수업 적용 흐름, 학습효과/운영 실효성
  - 문서화/재현성/배포(15) : README, 실행법, 예시 데이터/화면, 링크

## 8. 최종 제출 체크

- [ ] 수업 적용 시나리오 1페이지
- [ ] 개발 로그(프롬프트/이슈/해결 기록)
- [ ] GitHub 커밋 기록(최소 5회 이상 권장)
